# Path A Enrichment -- New Spec Section

## Executive Summary

Path A (`_run_task_subprocess()` at `executor.py:1064-1068`) is the production execution path for every properly structured tasklist generated by `sc-tasklist-protocol`. It currently sends a minimal 3-line prompt per task: task ID, phase file path, and a deliverables-only description. Path B (`build_prompt()` in `process.py:123-204`), a fallback for malformed files, sends a rich prompt with sprint context, scope boundary, skill invocation, and execution rules. No existing v3.7 spec section addresses this asymmetry.

This chunk defines 8 implementation tasks across 4 groups:

- **Group A** (PA-01 through PA-03): Per-task prompt enrichment adding ~280 tokens per subprocess -- task block extraction, scope boundary, and sprint context header. These three additions were validated by adversarial debate (context-02) and represent the minimum viable prompt for reliable task execution.
- **Group B** (PA-04 through PA-06): Three compounding bugs that zero out the TurnLedger reimbursement system. Fixing these activates an already-tested economic feedback loop with ~15 LOC total.
- **Group C** (PA-07): Evaluation and conditional wiring of `build_task_context()`, a fully implemented and tested function that is never called.
- **Group D** (PA-08): Post-task deliverable existence verification.

**Net impact**: ~120 LOC added, ~5 LOC modified, ~280 tokens added per task subprocess, zero new dependencies.

## Rationale

Path A is the production path. All production sprints using `sc-tasklist-protocol`-generated tasklists execute via Path A. Path B fires only for malformed or legacy phase files without `### T<PP>.<TT>` headings. Every improvement that the v3.7 spec targets at `build_prompt()` benefits only the fallback path. This section ensures the production path receives equivalent investment.

The adversarial debates (context-02) validated three prompt additions and rejected two. The rejections are architecturally sound: result file contracts are inferior to post-task hooks, and stop-on-STRICT-fail is unnecessary because Path A's subprocess isolation is itself the halt mechanism. The three approved additions are low-cost (~280 tokens total) and address validated deficiencies where worker behavior depends on inference rather than explicit instruction.

The TurnLedger bugs are not new findings but they have no existing spec coverage. They compound: `turns_consumed` returns 0, so reimbursement is always 0; `output_path` is never set, so the anti-instinct gate vacuously passes; `gate_rollout_mode` defaults to "off", so even if the other two were fixed, the gate would not run. Fixing all three activates an economic feedback loop that is already fully tested (50+ tests validate the math on non-zero inputs).

   def _extract_task_block(phase_file: Path, task_id: str) -> str:
       """Extract the raw markdown block for a single task from a phase file.

       Returns the heading line and all content up to the next task heading
       or EOF. Returns empty string if task_id is not found.
       """
       content = phase_file.read_text()

               block_end = headings[i + 1].start() if i + 1 < len(headings) else len(content)
               return content[block_start:block_end].rstrip()

       """Extract expected artifact paths from a task's markdown block.

       Parses **Artifacts (Intended Paths):** and **Deliverables:** sections
       for file paths. Resolves TASKLIST_ROOT placeholders against artifact_root.
       """
       paths = []

       for line in task_block.splitlines():
           stripped = line.strip()
           if stripped.startswith("**Artifacts") or stripped.startswith("**Deliverables"):

               path_str = stripped.lstrip("- ").strip("`")
               path_str = path_str.replace("TASKLIST_ROOT", str(artifact_root))
               paths.append(Path(path_str))
       return paths

### Risk Assessment

- **Group A** (PA-01 through PA-03): Low risk. Prompt-only changes. Workers receive more information but the subprocess boundary is unchanged. Rollback is prompt string deletion.
- **Group B** (PA-04 through PA-06): Low risk for PA-04/PA-05 (activating existing code paths). Low-medium risk for PA-06 (behavioral default change, but shadow mode is non-blocking by design).
- **Group C** (PA-07): Medium risk. Inter-task coupling is a design change. Mitigated by config flag (default off).
- **Group D** (PA-08): Low risk. Observability only, no enforcement.
