# Position B: Semantic Chunking with Hash Fingerprints

## Strategy

Split each roadmap into semantic chunks, fingerprint each chunk, then use a **two-pass approach**:
1. **Pass 1 (Pre-comparison)**: Hash each chunk from both files to identify identical/near-identical vs. different chunks
2. **Pass 2 (Selective storage)**: Store full content ONLY for chunks that differ; store hash-only stubs for matching chunks

## Compression Mechanics

### Step 1: Chunk Boundary Detection

Chunks are defined at the **phase level** and **top-level section level**:

```
Chunk boundaries for Opus:
  chunk-meta:     YAML frontmatter (lines 1-15)
  chunk-exec:     Executive Summary (lines 17-41)
  chunk-integ:    Integration Points (lines 43-58)
  chunk-phase-1:  Phase 1 table + header (lines 63-100)
  chunk-phase-2:  Phase 2 table + header (lines 104-139)
  chunk-phase-3:  Phase 3 table + header (lines 143-179)
  chunk-phase-4:  Phase 4 table + header (lines 183-223)
  chunk-phase-5:  Phase 5 table + header (lines 227-272)
  chunk-phase-6:  Phase 6 table + header (lines 276-318)
  chunk-risk:     Risk Assessment (lines 322-349)
  chunk-resource: Resources (lines 352-385)
  chunk-success:  Success Criteria (lines 388-430)
  chunk-timeline: Timeline (lines 433-473)

Chunk boundaries for Haiku:
  chunk-meta:     YAML frontmatter (lines 1-5)
  chunk-exec:     Executive Summary (lines 7-27)
  chunk-phase-1:  Phase 1 table + integration points (lines 29-82)
  chunk-phase-2:  Phase 2 table + integration points (lines 84-136)
  chunk-phase-3:  Phase 3 table + integration points (lines 138-205)
  chunk-phase-4:  Phase 4 table + integration points (lines 207-395)
  chunk-phase-5:  Phase 5 table + integration points (lines 397-530)
  chunk-phase-6:  Phase 6 table + integration points (lines 532-547)
  chunk-success:  Success criteria sections (lines 548-590)
  chunk-timeline: Timeline + gates (lines 592-663)
```

### Step 2: Hash Fingerprinting

For each chunk:
1. Normalize whitespace (collapse multiple spaces/newlines)
2. Normalize markdown formatting (strip bold/italic markers)
3. Compute SHA-256 hash of normalized content
4. Store: `{chunk_id, hash, line_count, word_count}`

### Step 3: Cross-File Comparison (Pre-Pass)

Compare chunk hashes between files:
- **Exact match**: Both files have a chunk with identical hash u2192 store hash-only stub
- **Near match (>85% similarity)**: Chunks cover same topic but differ in detail u2192 store both with similarity score
- **No match**: Chunk exists in only one file u2192 store full content

## Estimated Compression

**Critical insight**: Because these roadmaps are generated by DIFFERENT MODELS with different organizational strategies, the actual chunk overlap is very low.

Realistic assessment:
- chunk-meta: DIFFERS (Opus has 12 fields, Haiku has 3)
- chunk-exec: DIFFERS (completely different narrative)
- chunk-phase-1 through chunk-phase-6: ALL DIFFER (different task IDs, different row counts, different organization)
- chunk-risk: Opus-only (Haiku embeds risk in phase blocks)
- chunk-resource: Opus-only
- chunk-success: DIFFERS
- chunk-timeline: DIFFERS

**Expected chunk match rate: <5%**

This means:
- Opus: 60KB u2192 ~54-57KB (only 5-10% reduction)
- Haiku: 76KB u2192 ~68-72KB (only 5-10% reduction)
- Combined for diff: ~122-129KB (from 136KB) u2014 barely any improvement

**Best case** (if models produced similar output):
- 60KB u2192 ~18-30KB (50-70% reduction)
- But this is NOT the realistic case for multi-model roadmaps

## Strengths

1. **Full fidelity for differences** u2014 no lossy compression; every difference preserved verbatim
2. **Theoretically optimal** u2014 if files were very similar, this would eliminate all redundancy
3. **Chunk-level diff is semantic** u2014 differences are aligned to meaningful boundaries (phases, sections)
4. **Hash validation** u2014 can verify integrity of matched chunks
5. **Recoverable** u2014 combined with originals, full content can be reconstructed

## Weaknesses

1. **Requires pre-comparison** u2014 BOTH files must be available to compress EITHER one. This creates a cross-file dependency that contradicts the requirement to "compress each of them before feeding them to deef"
2. **Minimal compression for different-model outputs** u2014 chunk overlap is <5% when models organize differently
3. **Structural mismatch problem** u2014 Opus has 13 chunks, Haiku has 10 chunks with different boundaries. Phase-level chunks don't align 1:1 because Haiku embeds integration points inside phases while Opus keeps them separate
4. **Near-match detection is expensive** u2014 computing similarity beyond exact hash match requires tokenization and fuzzy comparison, adding complexity
5. **Not deterministic per-file** u2014 compression output depends on the OTHER file, so re-running with a different comparison target produces different compressed output
6. **Doesn't solve the actual size problem** u2014 the diff tool still sees ~90% of the original content because nothing matched

## Diff Behavior

The diff tool sees:
- HASH-ONLY stubs for matching chunks (rare in this case)
- Full content for differing chunks (almost everything)
- The actual diff is barely smaller than diffing the originals
- The chunk alignment provides SOME value by making the diff semantically structured

## Fatal Flaw for This Use Case

The user's specific problem is that the files are TOO LARGE for the diff tool. Position B's compression is contingent on file similarity. For roadmaps generated by different models (Opus vs Haiku) with fundamentally different organizational strategies, the overlap is negligible. This means Position B fails to solve the stated problem: the compressed files are still ~90% of original size.
