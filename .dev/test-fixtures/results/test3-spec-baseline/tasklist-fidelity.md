## Deviation Report

### DEV-001
- **ID**: DEV-001
- **Severity**: MEDIUM
- **Deviation**: Crypto review gate (Phase 1.5) is not enforced via task-level dependencies. Phase 2 tasks T02.01 and T02.02 can begin before T01.15 (security engineer code review) completes, because their dependency chains only reference T01.08 (JwtService) and T01.14 (DB pool), bypassing the crypto review gate.
- **Upstream Quote**: "**Exit Criteria**: All units pass. **Crypto review signed off.** Benchmark confirms ~250ms bcrypt timing and <50ms JWT sign/verify. Migrations run up and down cleanly. DI container wires correctly."
- **Downstream Quote**: T02.01: "**Dependencies:** T01.08, T01.14"; T02.02: "**Dependencies:** T02.01"
- **Impact**: In automated sprint execution, Phase 2 tasks could start before the crypto review gate is satisfied, potentially building stateful services atop unreviewed cryptographic foundations. The END-OF-PHASE CHECKPOINT exists as an informational gate but task-level dependencies don't enforce it.
- **Recommended Correction**: Add T01.15 (or T01.16) as a dependency to T02.01 so that the crypto review gate is structurally enforced in the dependency chain, not just documented in checkpoint text.

### DEV-002
- **ID**: DEV-002
- **Severity**: MEDIUM
- **Deviation**: Phase 1 exit criteria require "<50ms JWT sign/verify" benchmark validation, but no tasklist task includes a JWT timing benchmark. T01.11 (JwtService unit tests) covers functional correctness (sign/verify round-trip, expired rejection, tampered rejection) but has no timing assertion.
- **Upstream Quote**: "Benchmark confirms ~250ms bcrypt timing and **<50ms JWT sign/verify**."
- **Downstream Quote**: T01.11 Acceptance Criteria: "Valid token: sign then verify returns original payload / Expired token: verify returns null / Tampered token: verify returns null / Wrong algorithm: verify returns null" — '[MISSING]' for timing benchmark.
- **Impact**: The Phase 1 exit criteria cannot be fully validated because no task produces evidence of JWT sign/verify latency. The bcrypt benchmark is covered by T01.07, creating asymmetric coverage between the two crypto services.
- **Recommended Correction**: Add a timing assertion to T01.11 acceptance criteria: "JWT sign latency < 50ms; JWT verify latency < 50ms (benchmarked over 100+ iterations)". Alternatively, create a dedicated benchmark task analogous to T01.07's bcrypt timing test.

### DEV-003
- **ID**: DEV-003
- **Severity**: LOW
- **Deviation**: Roadmap Item IDs (R-001 through R-087) and Deliverable IDs (D-0001 through D-0087) are generated by the tasklist as sequential numbering of roadmap tasks. The roadmap itself contains no R-NNN or D-NNNN identifiers.
- **Upstream Quote**: Roadmap uses section numbering (Phase 1.1, 1.2, etc.) and task descriptions in tables — no R-NNN or D-NNNN identifiers anywhere in the document.
- **Downstream Quote**: T01.01: "Roadmap Item IDs | R-001"; "Deliverable IDs | D-0001" — and similarly for all 87 tasks.
- **Impact**: Minimal. The IDs are a consistent sequential mapping (R-001 = first roadmap task, R-087 = last) and every generated ID corresponds to a real roadmap task. Cross-referencing from tasklist back to roadmap requires content matching rather than ID lookup, but the 1:1 mapping is unambiguous.
- **Recommended Correction**: Either add R-NNN identifiers to the roadmap to enable bidirectional traceability, or add a mapping table to the tasklist index showing which roadmap section/row each R-NNN corresponds to.
