## 1. Package Layout

### Existing CLI Subpackage Structure

```
src/superclaude/cli/
├── __init__.py
├── main.py                   # Click group; registers all subcommand groups
├── sprint/                   # Sprint pipeline (reference implementation)
├── roadmap/                  # Roadmap pipeline
├── tasklist/                 # Tasklist pipeline
├── cleanup_audit/            # Cleanup-audit pipeline (closest portify peer)
├── pipeline/                 # Shared primitives: models.py, process.py, executor.py, gates.py
└── audit/                    # Static audit utilities (independent)
```

### Proposed cli_portify Subpackage

```
src/superclaude/cli/cli_portify/
├── __init__.py               # Exports cli_portify_group; marker: "Generated by" / "Portified from"
├── commands.py               # Click group registration + top-level CLI wiring
├── config.py                 # PortifyConfig (extends / wraps PipelineConfig)
├── models.py                 # Domain models: PortifyRun, TurnLedger, FailureType, etc.
├── process.py                # PortifyProcess (extends ClaudeProcess)
├── executor.py               # Sequential step executor for portify pipeline
├── gates.py                  # Gate criteria factories and PASS_NO_SIGNAL logic
├── prompts.py                # Prompt builder (synthesize_spec, analyze_workflow, etc.)
├── inventory.py              # Agent/skill/command discovery with warnings.warn()
├── contract.py               # Phase-contract schema management
├── monitor.py                # TUI/progress monitoring
├── logging_.py               # Structured logging (mirrors cleanup_audit/logging_.py)
├── diagnostics.py            # Diagnostic helpers
└── steps/                   # Optional: step-specific modules (synthesize_spec.py, etc.)
```

**No collision**: `cli_portify` is a new directory under `src/superclaude/cli/`. No existing file shares this name. The name does not conflict with any module in `sprint/`, `roadmap/`, `tasklist/`, `cleanup_audit/`, `pipeline/`, or `audit/`.

## 2. CLI Registration Pattern

`src/superclaude/cli/main.py` registers subcommand groups via:

```python
from superclaude.cli.<subpackage> import <group_name>
main.add_command(<group_name>, name="<cli-name>")
```

Existing registrations (lines 354–368 of main.py):
```python
from superclaude.cli.sprint import sprint_group
main.add_command(sprint_group, name="sprint")

from superclaude.cli.roadmap import roadmap_group
main.add_command(roadmap_group, name="roadmap")

from superclaude.cli.cleanup_audit import cleanup_audit_group
main.add_command(cleanup_audit_group, name="cleanup-audit")

from superclaude.cli.tasklist import tasklist_group
main.add_command(tasklist_group, name="tasklist")
```

**cli_portify registration** (to be added in Phase 10):
```python
from superclaude.cli.cli_portify import cli_portify_group
main.add_command(cli_portify_group, name="cli-portify")
```

The `main()` Click group uses `@click.group()` with no complex state; adding a new `add_command()` call is the standard, safe registration pattern.

## 3. Framework Coupling Points

### 3a. Base Type Imports

| Type | Module | Role in cli_portify |
|------|--------|---------------------|
| `PipelineConfig` | `superclaude.cli.pipeline.models` | Base config; `PortifyConfig` wraps or extends |
| `Step` | `superclaude.cli.pipeline.models` | Pipeline step definition |
| `StepResult` | `superclaude.cli.pipeline.models` | Step outcome record |
| `GateCriteria` | `superclaude.cli.pipeline.models` | Gate pass/fail criteria |
| `GateMode` | `superclaude.cli.pipeline.models` | BLOCKING vs TRAILING gate behavior |
| `SemanticCheck` | `superclaude.cli.pipeline.models` | Pure Python check applied to output |
| `ClaudeProcess` | `superclaude.cli.pipeline.process` | Subprocess lifecycle base; `PortifyProcess` extends |

### 3b. Import Graph (no circular imports)

```
cli_portify/* → pipeline.models  (data structures)
cli_portify/* → pipeline.process (ClaudeProcess base)
cli_portify/* → pipeline.executor (optional: reuse executor base)
cli_portify/* → pipeline.gates   (optional: reuse gate evaluation)
```

NFR-007 compliance: `pipeline/models.py` and `pipeline/process.py` have zero imports from `sprint/` or `roadmap/`. `cli_portify/` will likewise import only from `pipeline/` generics — never from `sprint/` or `roadmap/`.

### 3c. Reference Architecture: cleanup_audit

`cleanup_audit/` is the closest analog to `cli_portify/`. It:
- Has `__init__.py` with `# Portified from` / `# Generated by` markers
- Exports `cleanup_audit_group` (Click group)
- Contains `process.py`, `executor.py`, `gates.py`, `models.py`, `config.py`, `prompts.py`, `monitor.py`, `logging_.py`, `diagnostics.py`, `commands.py`

`cli_portify/` mirrors this layout exactly.

## 4. Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Subpackage location | `src/superclaude/cli/cli_portify/` | Consistent with cleanup_audit, sprint, roadmap |
| Registration | `main.add_command(cli_portify_group, name="cli-portify")` | Standard Click group pattern |
| Process base | `PortifyProcess(ClaudeProcess)` | Inherits `--tools default`, env isolation, signal handling |
| Config base | `PortifyConfig` wraps `PipelineConfig` | Adds portify-specific fields without modifying shared base |
| Sequential execution | Single-threaded step executor | Portify is inherently sequential; no parallelism needed |
| Gate enforcement | Adapts `pipeline.gates` GateCriteria | Reuses existing gate evaluation machinery |
