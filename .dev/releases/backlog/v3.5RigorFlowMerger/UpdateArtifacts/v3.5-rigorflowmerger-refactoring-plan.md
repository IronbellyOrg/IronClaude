## Cross-cutting refactoring decisions

These are the **global rules** the follow-up agent should apply before editing any improve-file.

### 1. v3.1 owns the canonical audit state machine
Do not introduce parallel audit-state vocabularies in shared models or sprint flow.

Use v3.1 concepts instead of creating competing enums like:
- `AgentHandoffState`
- generic `AuditPassState` in shared contracts
- ad hoc completion state vocabularies in `pipeline/models.py`

### 2. v3.1 owns the canonical gate failure taxonomy
Do not create a second first-class shared taxonomy for the same gate failures.

Prefer v3.1 `failure_class` semantics:
- `policy`
- `transient`
- `system`
- `timeout`
- `unknown`

Other taxonomies may exist only as **subclassification layers** if clearly scoped.

### 3. v3.1 owns deterministic gate/transition semantics
Do not restate broad generic gate rules in parallel form if they are already covered by:
- blocking-check failure => failed
- missing/unknown deterministic inputs => failed
- transition legality before completion/release

### 4. Task-unified and quality-agent proposals must not duplicate audit-gate ownership
If v3.1 added `sc-audit-gate` / `sc-audit-gate-protocol`, then task-unified routing rules should hand off audit enforcement to those layers rather than re-owning generic gate semantics.

### 5. Framework-generated markdown artifacts changed old heuristics
After v3.0/v3.05, `.md` files are no longer safely classifiable as “project-only” inputs because framework-generated gate artifacts now exist, including examples like:
- `wiring-report.md`
- `anti-instinct-audit.md`

### 6. Evidence discipline must extend, not duplicate
Where older improve-items call for stricter evidence requirements, retain only the incremental value that is not already covered by newer contracts.
