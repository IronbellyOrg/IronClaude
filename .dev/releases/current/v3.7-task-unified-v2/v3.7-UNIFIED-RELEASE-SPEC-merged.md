## 1. Release Overview

### Executive Summary

v3.7 task-unified-v2 delivers three interconnected improvements to the Sprint CLI pipeline:

1. **Checkpoint Enforcement** -- A three-layer defense architecture (Prevention, Detection, Remediation) that ensures checkpoint files are always written during sprint execution. Addresses a triple failure chain discovered during the OntRAG R0+R1 sprint (2026-03-22) where Phase 3 completed all tasks but failed to write either checkpoint file.

2. **Sprint TUI v2** -- A comprehensive visual UX refactor that surfaces rich real-time telemetry from Claude's stream-json output into the sprint dashboard. Adds 10 features across 7 files: activity stream, enhanced phase table, dual progress bars, conditional error panel, LLM context preview, post-phase summary generation with Haiku narratives, tmux summary pane integration, and release retrospective generation.

3. **Naming Consolidation** -- Resolves a three-layer naming collision where `/sc:task`, `task-unified.md`, and `sc:task-unified-protocol` each use different names for the same conceptual unit. Canonicalizes on `/sc:task` as the user-facing command.

## 2. Problem Statement

### 2.1 The Checkpoint Enforcement Failure

During the OntRAG R0+R1 sprint (2026-03-22, 7 phases), Phase 3 (Configuration & Utilities) completed all 6 tasks (T03.01-T03.06) and produced all 6 deliverables (D-0013 through D-0018) but **failed to write either checkpoint file** (CP-P03-T01-T05, CP-P03-END). This is the only phase out of 7 that missed its checkpoints. The failure went undetected until post-sprint manual inspection approximately 24 hours later.

**Root Cause: Triple Failure Chain**

Three independent safeguards were all absent simultaneously:

| # | Cause | File | Evidence |
|---|-------|------|----------|
| 1 (Primary) | Agent prompt contains no checkpoint instructions | `process.py` lines 169-203 | `build_prompt()` tells agent "After completing all tasks, STOP immediately" with zero mention of checkpoints |
| 2 (Structural) | Phase 3 uniquely has two checkpoint sections with non-task heading pattern | `phase-3-tasklist.md` lines 285-314 | `### Checkpoint:` heading is structurally distinct from `### T<NN>.<NN>` pattern recognized by task scanner |
| 3 (Pipeline) | No checkpoint enforcement or detection in sprint executor | `executor.py` lines 1592-1603 | `_check_checkpoint_pass()` exists but is only called as crash recovery heuristic, never as post-phase gate |

**Phase comparison showing the structural anomaly:**

| Phase | Task Headings | Checkpoint Headings | Total `###` Headings |
|-------|--------------|--------------------|--------------------|
| 1 | 5 | 1 | 6 |
| 2 | 6 | 1 | 7 |
| **3** | **6** | **2** | **8** |
| 4 | 4 | 1 | 5 |
| 5 | 5 | 1 | 6 |
| 6 | 5 | 1 | 6 |
| 7 | 4 | 1 | 5 |

**Why other phases succeeded:** The agents for phases 1, 2, 4, 5, 6, and 7 wrote checkpoints **voluntarily** -- they read the `### Checkpoint:` sections and inferred they should write files. This was emergent behavior, not instructed. Phase 3 differed due to: (a) the agent planning only T-numbered tasks, and (b) 744KB output (largest of all phases) creating context pressure.

**Impact:**
- No structured record that Phase 3's deliverables were verified
- No automated detection of the gap (discovered manually ~24 hours later)
- The 86% voluntary checkpoint write rate (6/7 phases) is unacceptable for deterministic pipeline execution

**Source**: `troubleshoot-missing-p03-checkpoint.md` Sections 1-4; `analysis-checkpoint-enforcement.md` Section 2

## 3. Solution Architecture

### 3.1 Checkpoint Enforcement -- Three-Layer Defense

The unified design implements checkpoint enforcement as three concentric, independent layers. Each layer is sufficient to catch the failure on its own; all three must fail simultaneously for a checkpoint to go permanently unwritten.

```
Layer 1 -- PREVENTION (Agent-Side)
  Solution 1: Prompt instructions tell the agent to write checkpoints
  Solution 2: Tasklist structure makes checkpoints indistinguishable from tasks
  -> Agent writes checkpoints as part of normal task execution

Layer 2 -- DETECTION (Executor-Side, Real-Time)
  Solution 3: Post-phase gate verifies checkpoint files exist on disk
  -> Sprint halts (or warns) if agent failed to write them

Layer 3 -- REMEDIATION (Post-Sprint, On-Demand)
  Solution 4: CLI verify + auto-recovery from evidence artifacts
  -> Missing checkpoints are detected, audited, and optionally regenerated
```

**Solution-to-Layer Mapping:**

| # | Solution Name | Layer | Root Cause Addressed | Mechanism |
|---|--------------|-------|---------------------|-----------|
| 1 | Prompt-Level Checkpoint Enforcement | Prevention | Cause 1 (primary) | Add `## Checkpoints` section to `build_prompt()` output |
| 2 | Tasklist-Level Checkpoint Normalization | Prevention | Cause 2 (structural) | Convert `### Checkpoint:` to `### T<PP>.<NN>` task entries |
| 3 | Post-Phase Checkpoint Enforcement Gate | Detection | Cause 3 (pipeline) | Executor verifies checkpoint files exist after phase PASS |
| 4 | Checkpoint Observability & Auto-Recovery | Remediation | Cause 3 + operational gap | Manifest tracking + CLI verify + auto-recovery from evidence |

**Solution-to-Root-Cause Coverage Matrix:**

| Solution | Cause 1 (No prompt) | Cause 2 (Structural mismatch) | Cause 3 (No enforcement) |
|----------|---------------------|-------------------------------|--------------------------|
| 1 (Prompt) | **FIXES** | Mitigates | Partially |
| 2 (Tasklist) | Not needed | **ELIMINATES** | Not addressed |
| 3 (Gate) | Not addressed | Not addressed | **FIXES** (detection) |
| 4 (Recovery) | Not addressed | Not addressed | **FIXES** (detection + remediation) |

**Minimum viable set**: Solution 1 + Solution 3 (or 4). This covers all three causes with prevention + detection.

**Full set**: All four solutions provide defense in depth where no single failure can result in a permanently missing checkpoint.

**Key Architectural Decision -- Shared Path Extraction Module:**

Both Solution 3 (per-phase gate) and Solution 4 (manifest/CLI) need to parse `Checkpoint Report Path:` from tasklists. Rather than duplicating this logic:
- `checkpoints.py` provides `extract_checkpoint_paths()` as a public function
- Solution 3's `_verify_checkpoints()` in executor.py imports from it
- Solution 4's `build_manifest()` imports from it

This eliminates the primary code duplication identified in the adversarial analysis.

#### New Modules

**`summarizer.py`** -- Three components:
1. `PhaseSummary` dataclass: Structured summary of a completed phase (tasks, files_changed, validations, reasoning_excerpts, errors, narrative)
2. `PhaseSummarizer` class: Orchestrates NDJSON extraction + Haiku narrative generation + markdown writing
3. `SummaryWorker` class (in `summarizer.py`): Background daemon thread pool -- one thread per completed phase, running in parallel with next phase execution. Exception handling wraps entire thread body so summary failure never affects sprint execution.

<!-- Source: Variant A, Section 4.5 -- merged per Change #3 -->
**Critical invariants**:
- Summary failure must never affect sprint execution. All exceptions caught.
- `SummaryWorker._summaries` dict MUST be guarded by `threading.Lock` -- unlike `OutputMonitor` (single writer thread), SummaryWorker uses a thread pool with concurrent writers.

**`retrospective.py`** -- Two components:
1. `ReleaseRetrospective` dataclass: Aggregated release-level summary (phase_outcomes, all_files, validation_matrix, all_errors, narrative)
2. `RetrospectiveGenerator` class: Reads all phase summaries, aggregates programmatically, generates Haiku narrative, writes `results/release-retrospective.md`

#### Haiku Narrative Pipeline

Both `PhaseSummarizer` and `RetrospectiveGenerator` follow the same pattern:
1. Check `shutil.which("claude")` is not None
2. Build prompt from structured data
3. Strip `CLAUDECODE` and `CLAUDE_CODE_ENTRYPOINT` from environment
4. Run: `claude --print --model claude-haiku-4-5 --max-turns 1 --dangerously-skip-permissions -p "<prompt>"`
5. Timeout: 30s
6. On failure: silently swallow exception, summary written without narrative section

## 15. Appendices

### Appendix A: Adversarial Analysis Summary (Checkpoint Enforcement)

The unified proposal includes a 6-criterion adversarial comparison of all 4 solutions.

**Criteria evaluated:**

| Criterion | Winner | Key Finding |
|-----------|--------|-------------|
| 1. Effectiveness | Solution 2 (VERY HIGH) | Drops failure rate to <1%; structural elimination vs behavioral reliance |
| 2. Implementation Complexity | Solution 1 (~60 LOC) | Lightest weight; Solution 4 heaviest at ~290 LOC |
| 3. Backward Compatibility | Solutions 1, 3, 4 (fully) | Solution 2 is a breaking change (requires tasklist regeneration) |
| 4. Defense in Depth | No single winner | Minimum set: Solution 1 + Solution 3/4. Full set: all four. |
| 5. Time to Value | Solution 1 (1-2 hours) | Fastest win; Solution 3 second (2-3 hours) |
| 6. Operational Overhead | Solution 1 (none) | Invisible to users; Solution 3 in full mode adds sprint halts |

**Complementary vs. Alternative determination:** All four solutions are **complementary, not alternative**. No solution is a strict replacement for another. The only partial overlap: if Solution 2 is deployed, Solution 1's prompt instructions become redundant for new tasklists but remain necessary for backward compatibility.

**Key finding:** "The redundancy between Solutions 1 and 2 is desirable (defense in depth), not wasteful."

**Redundancy eliminated:** `checkpoints.py` provides shared `extract_checkpoint_paths()` used by both Solution 3 (per-phase gate) and Solution 4 (manifest), avoiding duplicate tasklist parsing.

**Source**: `unified-checkpoint-enforcement-proposal.md` Part 1
