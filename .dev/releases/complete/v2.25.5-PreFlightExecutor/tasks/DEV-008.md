# DEV-008: Add --start/--end Phase Filter Interaction as Spec Acceptance Criterion

## Context

The spec-fidelity report identified that the roadmap adds OQ-004 (`--phases` filter interaction with preflight) as a resolved operational question, but the spec contains no mention of how phase-range filtering interacts with preflight execution. Without spec coverage, there are no acceptance criteria or required tests for this behavior.

## Evidence

- **Spec states**: No mention of `--start`/`--end` (phase-range) interaction with preflight anywhere in the spec. FR-PREFLIGHT.2 defines `execute_preflight_phases()` but does not specify that it must respect the user's phase-range selection.
- **Roadmap states**: "OQ-004 (--phases filter): Respect `--phases` for preflight phases too. If `--phases 2,3` excludes Phase 1, preflight skips it. Consistent behavior." (line 319)
- **Codebase truth**: The sprint CLI uses `--start` and `--end` options (not `--phases`), which populate `SprintConfig.start_phase` and `SprintConfig.end_phase`. The `active_phases` property at `src/superclaude/cli/sprint/models.py:324-327` filters phases to the `[start, end]` range. The roadmap's `execute_preflight_phases()` design already filters `config.active_phases` (roadmap line 87), which inherently respects the range. However, the absence of a spec acceptance criterion means no test is required to verify this behavior, creating a gap.
- **OQ-004 terminology note**: The roadmap says `--phases 2,3` but the actual CLI flag is `--start N`/`--end N` (a contiguous range). The roadmap's OQ-004 should use the correct flag names.

## Deviation Confirmed

This deviation is **confirmed valid**. The concern is architecturally real: if the preflight executor were implemented with a direct `config.phases` iteration instead of `config.active_phases`, it would bypass phase-range filtering and execute preflight phases the user explicitly excluded. The spec must mandate this behavior so implementers and tests enforce it.
