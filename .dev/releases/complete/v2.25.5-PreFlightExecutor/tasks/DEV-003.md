## Deviation

The roadmap (Phase 2) generifies the `empirical_gate_v1` classifier return values to `"pass"` / `"fail"`, contradicting the spec-defined labels `"WORKING"`, `"BROKEN"`, and `"CLI FAILURE"` with their exact conditional logic.

## Spec Contract (Authoritative)

From spec section 4.5:

```python
CLASSIFIERS: dict[str, ClassifierFn] = {
    "empirical_gate_v1": lambda exit_code, stdout, stderr: (
        "CLI FAILURE" if exit_code != 0
        else "WORKING" if "PINEAPPLE" in stdout
        else "BROKEN"
    ),
}
```

From spec section 5.2 (classifier signature):
> Returns classification label string (e.g., `"WORKING"`, `"BROKEN"`).

From spec section 8.1 (unit tests):
- `test_empirical_gate_v1_working`: exit_code=0, "PINEAPPLE" in stdout -> `"WORKING"`
- `test_empirical_gate_v1_broken`: exit_code=0, no "PINEAPPLE" -> `"BROKEN"`
- `test_empirical_gate_v1_cli_failure`: exit_code=1 -> `"CLI FAILURE"`

## Roadmap Contradiction

Roadmap Phase 2, step 3:
> Returns classification label (e.g., `"pass"`, `"fail"`)

This replaces the three-state classifier (`WORKING` / `BROKEN` / `CLI FAILURE`) with a two-state generic (`pass` / `fail`), losing the semantic distinction between a non-zero exit code failure and a zero-exit-code functional failure.

## Validated Addition

The roadmap's `"error"` label for classifier exception handling (OQ-003 / GAP-03) is a valid additive behavior not covered by the spec. This label applies only when the classifier function itself raises an exception, and should be retained alongside the spec-defined labels.

## Acceptance Criteria

- [ ] `empirical_gate_v1` in `classifiers.py` returns exactly `"CLI FAILURE"` when `exit_code != 0`
- [ ] `empirical_gate_v1` returns exactly `"WORKING"` when `exit_code == 0` and `"PINEAPPLE"` in `stdout`
- [ ] `empirical_gate_v1` returns exactly `"BROKEN"` when `exit_code == 0` and `"PINEAPPLE"` not in `stdout`
- [ ] Classifier exception wrapper returns `"error"` and logs at WARNING (additive from OQ-003)
- [ ] Unit tests validate all three spec-defined labels plus the `"error"` path
- [ ] Roadmap Phase 2 step 3 description updated to reference `"WORKING"`, `"BROKEN"`, `"CLI FAILURE"` labels instead of `"pass"`, `"fail"`
- [ ] No downstream code depends on `"pass"` / `"fail"` labels (verified by grep)

## Implementation Notes

- **Target file**: `src/superclaude/cli/sprint/classifiers.py` (new module, does not yet exist)
- **Test file**: `tests/sprint/test_classifiers.py` (per spec section 8.1)
- **No existing code to migrate**: The `FailureClassifier` in `diagnostics.py` is a separate post-failure diagnostic system (stall/timeout/crash/error/unknown) unrelated to the preflight classifier contract
- **Label semantics matter**: The three-state model captures a real distinction -- `CLI FAILURE` (the tool itself broke) vs `BROKEN` (the tool ran but produced wrong output) vs `WORKING` (tool ran and output matched expectations). Collapsing to `pass`/`fail` loses the `BROKEN` vs `CLI FAILURE` distinction that downstream decision-making may depend on.
