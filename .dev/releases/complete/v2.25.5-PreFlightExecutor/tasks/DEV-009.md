# DEV-009: Remove OQ-005 Ordering Constraint as Open Question -- Spec Already Resolves It

## Deviation Summary

The roadmap introduces OQ-005 as a "pending stakeholder input" open question about whether python-first execution ordering should be documented as a guaranteed contract. The spec considers this settled by design: six separate sections establish python-first ordering as a structural invariant of the preflight architecture, and Section 11 explicitly declares "None. All design questions resolved."

**Confirmed findings:**

1. **Spec Section 2 (Solution Overview):** States preflight "marks those phases as complete before the main Claude-subprocess loop begins" -- this is the feature's core design, not a policy option.

2. **Spec Section 2.1 (Key Design Decisions):** Row "Guard location" explicitly chose "Pre-sprint hook in `execute_sprint()`" over "Per-phase guard in loop" with rationale: "Pre-sprint is cleanest: all preflight work completes before any Claude subprocess launches."

3. **Spec Section 2.2 (Workflow):** The data flow diagram shows `[PRE-SPRINT HOOK] execute_preflight_phases(config)` unconditionally preceding the main loop. There is no conditional or configurable ordering path.

4. **FR-PREFLIGHT.2:** Acceptance criterion states "Called at the top of `execute_sprint()`, before the phase loop" -- this is a hard requirement, not a suggestion.

5. **FR-PREFLIGHT.7:** States "Phases with `execution_mode == 'python'` are skipped in the main loop (already executed in preflight)" -- this structurally depends on preflight running first.

6. **Spec Section 11 (Open Items):** "None. All design questions resolved through adversarial debate pipeline (Q4-Q10)." -- The spec author explicitly closed all open questions.

7. **Appendix A (Glossary):** Defines "Preflight" as "Pre-sprint execution of python-mode phases before any Claude subprocess launches."

The roadmap's OQ-005 contradicts the spec by marking a resolved design decision as "Pending stakeholder input." This could delay implementation or cause implementers to hedge the ordering logic rather than implementing the spec's clear architecture.

**Nuance:** The roadmap's concern about "not documenting as a guaranteed contract" has a narrow validity regarding user-facing documentation. However, this is a documentation-level decision, not an implementation-level one. The implementation must be python-first per the spec. Whether ordering is promoted as a public API guarantee can be decided separately and does not require blocking the feature.

## Implementation Requirement

Update the roadmap's Open Questions Resolution table to close OQ-005 as resolved by the spec:

1. **Change OQ-005 status** from "Pending stakeholder input" to "Resolved"

2. **Replace OQ-005 resolution text** with:
   > Resolved by spec design. The spec's architecture (SS2, SS2.1, SS2.2, FR-PREFLIGHT.2, FR-PREFLIGHT.7) establishes python-first ordering as a structural invariant -- preflight phases run before the main Claude loop unconditionally. This is not a policy choice; it is the feature's core mechanism. User-facing documentation of ordering guarantees is a separate, non-blocking concern.

3. **Remove the "Pending stakeholder input" qualifier** from the Executive Summary line: "OQ-001 through OQ-004 resolved upfront; OQ-005 (ordering semantics) flagged for stakeholder confirmation." Replace with: "OQ-001 through OQ-005 resolved upfront."

4. **Do NOT change the implementation** -- the roadmap's Phase 4 already correctly describes calling `execute_preflight_phases()` before the main loop, which is the spec-mandated behavior.
