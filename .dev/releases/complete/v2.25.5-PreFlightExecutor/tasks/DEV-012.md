## Roadmap / Test Strategy Discrepancies

### 1. Test strategy inflates integration count

The test strategy (VM-3 + VM-4) lists 12 integration tests total (5 in VM-3, 7 in VM-4), not the spec's 8. Four additional integration tests were introduced:

- T-4.2: Zero `ClaudeProcess` instantiation for python-mode
- T-4.4: Nested `claude --print` via subprocess (no deadlock)
- T-4.6: Single-line rollback verification
- T-4.7: `--phases 2,3` filter excludes preflight Phase 1 (from OQ-004)

These are valid additions from adversarial review but are not in the spec's SS8.2 table.

### 2. Classifier test labels conflict (inherits DEV-003)

Test strategy VM-2 describes:
- T-2.1: `empirical_gate_v1(0, "ok", "")` -> `"pass"`
- T-2.2: `empirical_gate_v1(1, "", "error")` -> `"fail"`

The spec (SS8.1) defines three distinct tests with labels `"WORKING"`, `"BROKEN"`, `"CLI FAILURE"`. The test strategy collapses three spec tests into two and uses wrong labels. (This is a downstream effect of DEV-003.)

### 3. Missing-command validation test not in spec

Test strategy T-1.8 ("Python-mode task with empty command raises `ClickException`") is from DEV-004's additive validation. Not in spec SS8.1.

### 4. Round-trip test not in spec

Test strategy T-1.9 ("Round-trip: write tasklist-index.md with modes -> parse -> verify") is additive. Not in spec SS8.1.

### 5. Test file paths differ (inherits DEV-007)

Spec uses `tests/sprint/{test_models.py, test_config.py, test_classifiers.py}` (3 files). Roadmap/test-strategy uses `tests/cli/sprint/test_preflight.py` (1 combined file). Existing project structure has tests at `tests/sprint/` (no `tests/cli/sprint/`).

## Impact Assessment

**Severity: LOW** -- The aggregate counts match (14 + 8 = 22), and all spec-required test scenarios are covered across the roadmap phases even if names and groupings differ. No functional impact if all spec-required test scenarios are ultimately implemented.

However, without explicit cross-referencing, an implementer could:
- Miss the three-way classifier test split (writing 2 tests instead of 3)
- Omit a spec-listed integration test that maps differently in the test strategy
- Double-count additive tests against the spec's totals
