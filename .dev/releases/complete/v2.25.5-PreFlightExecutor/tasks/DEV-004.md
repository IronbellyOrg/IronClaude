# DEV-004: Reconcile empty-command validation for python-mode tasks

## Problem

The roadmap (Phase 1, step 7) adds an explicit validation rule not present in the spec: if a phase has `execution_mode == "python"` and any task has `command == ""`, raise `click.ClickException`. The spec (FR-PREFLIGHT.3) allows empty commands on tasks without specifying what happens when a python-mode phase contains such tasks.

This is a spec gap, not a roadmap error. Without this validation, `shlex.split("")` returns `[]` and `subprocess.run([], ...)` would raise a runtime error with a non-actionable traceback. The roadmap's fail-fast validation is semantically correct and defensively sound.

## Deviation Evidence

**Spec Quote (FR-PREFLIGHT.3):**
> "Tasks without a `**Command:**` field have `command == ''`"

**Roadmap Quote (Phase 1, step 7):**
> "Add explicit validation for python-mode tasks with empty commands -- If a phase has `execution_mode == 'python'` and any task has `command == ''`, raise `click.ClickException(f"Task {task_id} in python-mode phase has no command")`"

**Spec gap:** FR-PREFLIGHT.2 acceptance criteria assume all python-mode tasks have commands (AC-5: "extracts the shell command", AC-6: "Executes each command via subprocess.run()") but never explicitly validate this precondition.

## Resolution

Add an acceptance criterion to FR-PREFLIGHT.2 in the spec to formalize this validation:

**In file:** `.dev/releases/backlog/v2.25.5-PreFlightExecutor/sprint-preflight-executor-spec.md`

**Location:** FR-PREFLIGHT.2, after existing acceptance criterion "For each task, extracts the shell command from the `**Command:**` markdown field" (AC-5).

**New acceptance criterion:**
```
- [ ] If any task in a python-mode phase has `command == ""`, raise `click.ClickException` with the task ID and phase number
```

**Rationale:** This formalizes a parse-time precondition that the spec's execution flow implicitly requires. It converts a runtime `subprocess.run([])` crash into an actionable error message identifying the offending task.

## Acceptance Criteria

- [ ] FR-PREFLIGHT.2 in the spec contains an explicit acceptance criterion for empty-command validation in python-mode phases
- [ ] The acceptance criterion specifies raising `click.ClickException` with both task ID and phase number
- [ ] The roadmap's Phase 1 step 7 aligns with the updated spec criterion (no further roadmap changes needed -- the roadmap already has this correct)
- [ ] A unit test name is added to spec section 8.1 for this validation (e.g., `test_python_mode_empty_command_raises`)
