## NEW FINDINGS

### C-111: Redundancy guard `tdd_file=None` override lost after `_build_steps` returns
**Severity:** IMPORTANT | **Tag:** [TDD/PRD]
**File:** `src/superclaude/cli/roadmap/executor.py` L861-867
**Bug:** When the primary input is a TDD and `--tdd-file` is also passed, `_build_steps` correctly logs a warning and does `config = dataclasses.replace(config, tdd_file=None)`. But this creates a LOCAL config inside `_build_steps` -- the caller (`execute_roadmap` at L1848) still has the original config with `tdd_file` set. When `_save_state(config, results)` runs, it saves the redundant `tdd_file` to the state file. On subsequent `--resume` or `tasklist validate`, the state file's `tdd_file` triggers auto-wire behavior with a file that was supposed to be ignored.

This is distinct from C-84 (which covers the dead auto-detection code). C-84's recommended fix ("keep only the redundancy guard") would preserve this bug.

**Required Fix:** Move the redundancy guard from `_build_steps` to `execute_roadmap`, after the `input_type` resolution and before `_build_steps()` call:
```python
# After L1823 (input_type resolution):
if config.input_type == "tdd" and config.tdd_file is not None:
    _log.warning("Ignoring --tdd-file: primary input is already a TDD document.")
    config = dataclasses.replace(config, tdd_file=None)
```

### C-113: _build_steps `effective_input_type` variable diverges from `config.input_type`
**Severity:** IMPORTANT | **Tag:** [TDD/PRD]
**File:** `src/superclaude/cli/roadmap/executor.py` L852-862
**Bug:** After the C-62 fix, `execute_roadmap` resolves `config.input_type` before calling `_build_steps`. But `_build_steps` still creates a local `effective_input_type = config.input_type` (L853) and uses it for the redundancy guard (L862). After C-84's fix removes the dead resolution code, `effective_input_type` becomes a pointless alias for `config.input_type`. But if C-84 is fixed carelessly (removing only the `if` block at L854-859 but leaving `effective_input_type`), the redundancy guard at L862 still uses the local variable. A future developer might change one and not the other.

This is a readability/maintenance trap, not a correctness bug today. But it becomes a correctness bug if someone reintroduces logic that modifies `effective_input_type` without also modifying `config`.

**Required Fix:** When implementing C-84, also eliminate the `effective_input_type` local variable. The redundancy guard should read `config.input_type` directly (or better yet, move the guard to `execute_roadmap` as recommended in C-111).
