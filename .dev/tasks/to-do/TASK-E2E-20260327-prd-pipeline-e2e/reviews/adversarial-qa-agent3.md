### E-01 [CRITICAL] — spec_parser requirement ID regex silently drops compound IDs (FR-AUTH-001, NFR-PERF-001)

**File:** `src/superclaude/cli/roadmap/spec_parser.py:299-305`

**Evidence:** The `_REQUIREMENT_PATTERNS` dict uses these regexes:
```python
"FR": re.compile(r'\bFR-\d+(?:\.\d+)?\b'),
"NFR": re.compile(r'\bNFR-\d+(?:\.\d+)?\b'),
```

These match only `FR-NNN` or `FR-N.N` formats. They do **not** match:
- `FR-AUTH-001` (TDD fixture uses this format -- 5 FRs)
- `NFR-PERF-001`, `NFR-REL-001`, `NFR-SEC-001` (TDD fixture uses these -- 4 NFRs)
- `FR-AUTH.1` (spec fixture and PRD fixture use this format -- 5 FRs)
- `NFR-AUTH.1` (spec and PRD fixtures use this format -- 3 NFRs)

Verified with Python regex testing: all compound-ID formats return empty matches.

**Impact:** The `check_signatures` structural checker uses `spec_parsed.requirement_ids` to detect phantom IDs in the roadmap. When the spec uses compound IDs, the `spec_ids` set is **empty**, which means:
1. Every roadmap reference to spec IDs becomes a "phantom ID" false positive (HIGH severity finding per `SEVERITY_RULES`)
2. No genuine phantom IDs are detected (false negatives)
3. All three test fixtures (`test-tdd-user-auth.md`, `test-spec-user-auth.md`, `test-prd-user-auth.md`) use compound ID formats, so **every test run will produce wrong results**

**Required Fix:** Expand the FR/NFR regex patterns to accept an optional alpha-segment:
```python
"FR": re.compile(r'\bFR-(?:[A-Z]+-)?\\d+(?:\\.\\d+)?\\b'),
"NFR": re.compile(r'\bNFR-(?:[A-Z]+-)?\\d+(?:\\.\\d+)?\\b'),
```
Also add a test that verifies compound IDs are extracted from the TDD fixture.

### E-02 [CRITICAL] — certify_prompts.py parser regex rejects all structural checker finding IDs

**File:** `src/superclaude/cli/roadmap/certify_prompts.py:258-272`

**Evidence:** `parse_certification_output()` uses:
```python
pattern = re.compile(r"(F-\d+)\s*:\s*(PASS|FAIL)\s*--\s*(.+)", re.IGNORECASE)
```

This only matches `F-NNN` IDs (e.g., `F-01`, `F-02`). However, the actual finding IDs generated by `structural_checkers.py:_make_finding()` use the format `{dimension}-{mismatch_type}-{stable_id[:8]}` -- e.g., `signatures-phantom_id-abc12345`, `nfrs-threshold_contradicted-def67890`.

Verified with Python regex testing: structural checker IDs produce no matches.

**Impact:** After remediation runs and the certification agent reports results, `parse_certification_output()` returns an **empty list** -- no findings are recognized as verified. The `route_certification_outcome()` then receives `total=0`, returning `certified=False` (the `total > 0` guard fails). This means certification always fails with 0 findings verified, even if every fix was correctly applied.

**Required Fix:** Either (a) update the regex to match the actual `{dimension}-{mismatch_type}-{hash}` format, or (b) normalize finding IDs to `F-NNN` format in the certification prompt and verify the LLM echoes them back in that format.

### E-03 [CRITICAL] — DIMENSION_SECTION_MAP is hardcoded to release-spec section numbering; TDD sections invisible to structural checkers

**File:** `src/superclaude/cli/roadmap/spec_parser.py:515-521`

**Evidence:**
```python
DIMENSION_SECTION_MAP: dict[str, list[str]] = {
    "signatures": ["FR-1", "FR-2", "3. Functional Requirements"],
    "data_models": ["4.1", "4.2", "4.3", "4. File Manifest"],
    "gates": ["FR-7", "FR-8", "5.1"],
    "cli": ["5.1", "CLI", "Commands"],
    "nfrs": ["6.", "NFR", "Non-Functional"],
}
```

The TDD fixture uses a different section numbering:
- Functional requirements are in **Section 5** ("5. Technical Requirements"), not section 3
- Data models are in **Section 7** ("7. Data Models"), not section 4
- Architecture is in **Section 6**, not mapped at all
- API specs are in **Section 8**, not mapped at all
- There is no "File Manifest" section; the TDD has "4. Success Metrics"

When `_get_sections_for_dimension()` uses this map to route TDD sections to checkers, it returns **empty** section lists for dimensions that have no matching headings. The checkers then operate on empty section content, producing false negatives (missed deviations).

**Impact:** For TDD input, the structural dimension checkers `check_signatures`, `check_gates`, and `check_cli` effectively skip all content-specific analysis because `_get_sections_for_dimension()` finds no matching sections. Only the full-text searches (e.g., `roadmap_full_text` substring checks) still work. Section-scoped analysis is completely blind to TDD content.

**Required Fix:** Either make `DIMENSION_SECTION_MAP` input-type-aware (with separate entries for TDD vs spec section numbering) or add TDD heading patterns as additional entries:
```python
"signatures": ["FR-1", "FR-2", "3. Functional Requirements", "5. Technical Requirements", "5.1"],
"data_models": ["4.1", "4.2", "4.3", "4. File Manifest", "7. Data Models", "7.1"],
```

### E-04 [IMPORTANT] — Fingerprint extraction ignores file paths, API endpoints, and data model field names

**File:** `src/superclaude/cli/roadmap/fingerprint.py:57-118`

**Evidence:** `extract_code_fingerprints()` only extracts three categories:
1. Backtick-delimited identifiers (`AuthService`, `TokenManager`)
2. Code block function/class definitions (`def login()`)
3. ALL_CAPS constants

It does NOT extract:
- File paths (`src/auth/auth-service.ts`) -- these are design artifacts critical to TDD fidelity
- API endpoints (`/auth/login`, `/auth/refresh`) -- also critical
- Data model field names from interface definitions (`displayName`, `lastLoginAt`)

The backtick extraction partially covers some of these (e.g., `AuthService` appears in backticks), but file paths and endpoint routes are typically not backtick-wrapped.

**Impact:** For TDD documents, fingerprint coverage ratio is artificially high because the denominator (total fingerprints) excludes entire classes of TDD-critical design artifacts. A roadmap could omit all file paths and API endpoints but still pass the fingerprint gate.

**Required Fix:** Add extraction categories for file paths (reuse `_FILE_PATH_RE`) and API endpoint patterns (`/auth/...`).

### E-05 [IMPORTANT] — _restore_from_state mutates a frozen dataclass via direct attribute assignment

**File:** `src/superclaude/cli/roadmap/executor.py:1725, 1742, 1751, 1762`

**Evidence:** Lines 1725 and 1742 mutate `config` directly:

```python
config.agents = restored    # line 1725
```

While `RoadmapConfig` is a `@dataclass` (not frozen), the function signature returns `RoadmapConfig` and earlier in the function (line 859) uses `dataclasses.replace()` for immutability. The mixing of mutation and `dataclasses.replace()` within the same module is an inconsistency that C-07 partially noted but didn't cover the tdd_file/prd_file mutations.

Note: C-07 already covers `config.agents` and `config.depth` mutation. This extends the finding to `config.tdd_file` and `config.prd_file` on lines 1751 and 1762, which were added as part of the TDD integration and use the same direct-mutation pattern.

**Impact:** If the caller holds a reference to the original config, the mutations propagate unexpectedly. The `dataclasses.replace()` pattern used in `_build_pipeline()` does not propagate back here.

**Required Fix:** Use `dataclasses.replace()` consistently for all config mutations in `_restore_from_state()`.

### E-09 [MINOR] — Test fixture TDD has inconsistent data model field naming between TypeScript interface and field table

**File:** `.dev/test-fixtures/test-tdd-user-auth.md`, Section 7 (Data Models)

**Evidence:** The `UserProfile` TypeScript interface (lines 372-380) uses camelCase:
```typescript
interface UserProfile {
  displayName: string;
  createdAt: string;
  updatedAt: string;
  lastLoginAt: string;
```

But the spec fixture (`test-spec-user-auth.md`) defines the same data model using Python-style snake_case in its interfaces (lines 180-190):

The structural checker `check_data_models` performs case-insensitive text matching (`roadmap_full_text = roadmap_text.lower()`), so `displayName` and `display_name` both lowercase to `displayname` and `display_name` respectively -- these are **different** lowercase strings. A roadmap generated from the spec fixture would use `display_name` which would NOT match a roadmap generated from the TDD fixture's `displayName`.

**Impact:** If spec and TDD are used together (spec + `--tdd-file`), the structural checker could report false positives on field name mismatches that are actually just naming convention differences between TypeScript and Python.

**Required Fix:** Either normalize underscore/camelCase during comparison, or ensure test fixtures use consistent naming conventions, or document that cross-format field matching requires normalization.
