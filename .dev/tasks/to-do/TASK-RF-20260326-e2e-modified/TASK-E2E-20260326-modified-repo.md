# E2E Pipeline Tests — TDD and Spec Paths in Modified Repo

## Task Overview

This task executes two end-to-end pipeline tests using `superclaude roadmap run` to verify that the TDD extraction path works correctly and that the existing spec extraction path remains unbroken after our modifications. Test 1 creates a populated TDD fixture for a "User Authentication Service" and runs the full pipeline, verifying auto-detection routes to `build_extract_prompt_tdd()` and that extraction.md contains all 14 body sections including 6 new TDD-specific sections. Test 2 creates a populated spec fixture for the same feature and runs the full pipeline, verifying auto-detection routes to `build_extract_prompt()` and that extraction.md contains only the standard 8 body sections with no TDD content leaking through.

All testing prior to this was unit-level (individual functions, string assertions, config defaults). This is the first time the actual CLI pipeline is exercised end-to-end with real documents. The pipeline spawns Claude subprocesses for each step, so each test run takes 30-60 minutes.

## Key Objectives

- Verify `detect_input_type()` correctly identifies TDD vs spec documents at runtime via auto-detection
- Confirm `build_extract_prompt_tdd()` produces extraction.md with all 14 body sections and 19 frontmatter fields
- Confirm all pipeline gates (EXTRACT through SPEC_FIDELITY) pass for TDD-derived output
- Confirm the spec path produces standard 8-section extraction with no TDD content leaks
- Confirm both pipelines produce valid roadmaps with fingerprint_coverage >= 0.7
- Compare Test 1 and Test 2 outputs to verify structural differences match expectations
- Document all results in a verification report
